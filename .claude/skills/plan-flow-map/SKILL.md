# plan-flow-map

플랜 문서를 다크모드 팔레트의 **리치 스크롤 문서**로 시각화한다.
타입별 블록(콜아웃·비교표·엔드포인트 카드·엔티티 카드·구현순서 표·파일 트리·코드)을 위→아래로 렌더링한다.

## 목적 — 긴 문서를 "빠르게 이해"시키는 도구

이 스킬의 목적은 길게 쓰인 마크다운 문서를 **읽기 쉽게** 보여주는 것이다. 보고서처럼 딱딱하게 원문을 그대로
옮기지 말고, 이해가 빠르도록 재구성한다. 단, 자유롭게 하면 에이전트마다 결과가 갈리고 원문이 누락되므로
**재구성은 반드시 `references/reshaping-rubric.md`의 규칙을 따른다** — 콘텐츠→블록 매핑 표, 무결성(누락
금지), 일관된 읽기 처리, 자가 점검. 이 규칙이 일관성과 완전성을 동시에 보장한다.

**출력은 2-레이어다:** 맨 위에 항상 펼쳐진 **한눈 요약(hero)** + 그 아래 **기본 접힘 상세 섹션**(펼쳐서
리뷰). 빠른 이해(hero)와 무손실 상세(접이식)를 동시에 준다. 요약은 세부를 *삭제*하는 게 아니라 위에 *얹는다*.

## ⛔ 최우선 규칙

- **전달받은 플랜 문서가 유일한 진실**이다. 문서에 없는 엔드포인트·필드·파일·게이트를 지어내지 않는다.
  실제 이름·경로·명령을 쓰고, 불확실하면 넣지 않는다.
- **어떤 마크다운이 와도 하나의 블록 문서로 재구성**한다(스키마는 `references/doc-schema.md`).
  출력은 항상 일관된 형태. 정해진 입력 템플릿을 요구하지 않는다.
- **HTML/CSS를 손으로 만들지 않는다.** 블록 JSON만 만든다 — `build.py`가 다크 스타일 HTML로 조립한다.
- **속도: 병렬 하위 에이전트(지원 시).** 플랜을 섹션별로 나눠 병렬 생성 후 병합한다(아래 Step 2).
  병렬 하위 에이전트가 없는 런타임에선 **순차로** 처리한다 — 산출물은 동일, 속도만 차이.

## 경로 규약 (런타임 중립)

- `<skill>` = **이 SKILL.md가 있는 디렉토리**(스킬 루트). 아래 모든 상대경로·명령은 이 폴더 기준이다.
- **절대경로를 하드코딩하지 않는다** — 런타임마다 설치 위치가 다르다: Claude Code `~/.claude/skills/`,
  Codex `~/.agents/skills/`, OpenCode는 둘 다 스캔. 하위 에이전트에 경로를 넘길 때는 메인이 `<skill>`을
  실제 절대경로로 치환해 전달한다.
- 실행은 되도록 `<skill>` 폴더를 기준으로 한다(`python3 scripts/build.py …` 형태 권장).

## 이 스킬의 파일

- `assets/template.html` — 렌더링 셸(CSS+테마 토글, 다크/라이트). **직접 수정 금지.**
- `scripts/build.py` — `doc.json`(블록 목록)을 HTML로 조립·주입. 블록별 렌더러 포함. (python3 표준 라이브러리만)
- `references/doc-schema.md` — `doc.json`/블록 스키마(11종 블록 형태). **작업 전 반드시 읽는다.**
- `references/reshaping-rubric.md` — 마크다운→블록 재구성 규칙(매핑 표·무결성·자가점검). 블록을 만드는
  모든 하위 에이전트가 **반드시 읽고 따른다.**

## 절차

중간 파일(`blocks/`, `doc.json`)은 임시 작업 폴더에 만든다(런타임이 주는 스크래치 디렉토리, 없으면
`mktemp -d` 또는 repo-local `.plan-flow-map-tmp/`; 이하 `<tmp>`). **최종 HTML은 `<skill>/cache/`에 캐시로
저장한다** — 게시/아티팩트 도구로 올리지 말고 파일로만 남긴다.

### Step 0 — 스키마 숙지
`references/doc-schema.md`를 읽는다(11개 블록 타입 + 어떤 콘텐츠에 어떤 블록을 쓰는지).

### Step 1 — 플랜 파악 + 섹션 분할
- 플랜 문서를 읽는다(경로가 주어지면 그 파일, 없으면 대화 컨텍스트의 플랜).
- `meta`를 정한다: `title`, `eyebrow`(예: "구현 계획"), `subtitle`(한두 문장 요약), `source`(문서 경로
  + git 레포면 `git rev-parse --short HEAD`·branch).
- 문서를 **순서대로 섹션 N개**로 나눈다(예: 목표·완료조건 / 공개계약·저장경계 / API 계약 / 구현순서 /
  파일범위 / 승인기준). 각 섹션의 **원문 텍스트 조각**을 확보한다.

### Step 2 — 섹션별 블록 JSON 생성 (병렬 우선, 순차 폴백)  ★속도 핵심
`<tmp>/blocks/` 를 만든다. **섹션마다 하위 에이전트 1개를 한 번에 병렬 스폰**한다 — 런타임의 병렬 하위
에이전트 기능을 쓴다(Claude Code=Task/Agent, OpenCode=`task` 툴, Codex=agents). **병렬을 지원하지 않으면
섹션을 순차로 하나씩 처리**한다(산출 동일). 순서 보존을 위해 출력 파일명에 2자리 인덱스를 준다:
`01_goal.json`, `02_contracts.json`, `03_api.json` … 각 하위 에이전트(또는 순차 처리)에 다음을 전달한다:

```
- 먼저 두 파일을 읽어라(경로는 메인이 <skill> 절대경로로 치환해 전달):
  <skill>/references/doc-schema.md      (블록 11종의 형태)
  <skill>/references/reshaping-rubric.md (재구성 규칙 — 반드시 준수)
- 담당 섹션 원문: <해당 섹션 마크다운 텍스트만>
- 할 일: 이 섹션을 블록 JSON "배열"([ {type,...}, ... ])로 재구성한다. reshaping-rubric의 §1 매핑 표를
  그대로 적용하고(같은 형태 → 같은 블록), §2 무결성(원문 항목 누락 0 · 기술 토큰 verbatim · 순서 유지),
  §3 읽기 처리(heading→한 줄 리드 prose→세부)를 지킨다. 문서에 없는 것은 지어내지 마라.
- 첫 블록은 이 섹션 제목의 {type:"heading"}.
- 파일 저장 직전 rubric §4 자가 점검을 수행하라(원문의 모든 문단·항목·표행·코드·엔드포인트·필드·경로가
  블록에 있는지). 빠진 것이 있으면 추가한 뒤 저장한다.
- 출력: 결과 JSON 배열을 <tmp>/blocks/<NN>_<name>.json 파일로 저장하라. 배열만, 주석/설명 없이.
- 반환: 파일 경로 + 담당 섹션에서 다룬 항목 수(문단/리스트/표행/엔드포인트/필드 등) 한 줄.
```

작은 플랜(섹션 1~2개)이면 하위 에이전트 없이 직접 만들어도 된다(스폰 오버헤드 > 이득).

### Step 2.5 — 한눈 요약(hero) 합성  ★2-레이어 상단
섹션 처리와 함께(병렬 지원 시 같이 스폰, 아니면 이어서) 합성 에이전트 1개가 **문서 전체**를 읽고 아래 블록
배열을 `<tmp>/blocks/00_summary.json`(00_ 이라 병합 시 맨 앞)에 저장한다:
```
[ {"type":"heading","level":2,"text":"한눈 요약","summary":true},   // summary:true → 접히지 않는 hero
  {"type":"callout","tone":"goal","title":"목표","body":"<1~2문장>"},
  {"type":"bullets","title":"핵심 변경","items":[<3~5개>]},
  {"type":"stats","items":[{"value":"..","label":".."}, ...]},        // 규모 칩(파일 수·배치 등)
  {"type":"callout","tone":"gate","title":"핵심 게이트","body":"<압축>"},
  {"type":"callout","tone":"warn","title":"최상위 위험","body":"<1~3개>"} ]
```
요약이지만 원문에 있는 사실만 쓴다(지어내기 금지). 슬롯(목표/변경/규모/게이트/위험)은 어떤 플랜에도 일반화된다.

### Step 3 — 병합 + 누락 점검
`<tmp>/blocks/*.json` 을 **파일명 순서대로** 읽어 배열을 이어붙인다(`00_summary`가 맨 앞 = hero 상단).
각 파일이 유효한 JSON 배열인지 확인(깨졌으면 해당 섹션만 다시 처리한다).

**누락 점검(무결성):** 원본 문서를 다시 훑어 각 섹션의 항목이 블록에 반영됐는지 대조한다 — 표 행 수,
엔드포인트 수, 필드 수, 파일 경로가 원문과 일치해야 한다. 빠진 섹션·항목이 있으면 해당 섹션만
`reshaping-rubric.md`를 다시 강조해 재처리한다. (이 대조는 메인 에이전트가 전체 원문+전체 블록을 갖고 수행.)

`meta` + 합친 `blocks`로 `<tmp>/doc.json`을 완성한다:
```jsonc
{ "meta": { "title": "...", "eyebrow": "...", "subtitle": "...", "source": "..." },
  "blocks": [ /* 섹션 순서대로 이어붙인 모든 블록 */ ] }
```

### Step 4 — 빌드 (스킬 경로에 캐시로 저장)
플랜의 짧은 슬러그를 정한다(원본 파일명 또는 주제 기반, kebab-case ascii — 예: `office-transition`).
최종 HTML과 그 doc.json을 `<skill>/cache/`에 쓴다. **게시/아티팩트 도구로 올리지 않는다(파일만).**
```bash
# <skill> = 이 SKILL.md가 있는 디렉토리의 절대경로, <tmp> = 임시 작업 폴더, <slug> = 위 슬러그
mkdir -p "<skill>/cache"
cp "<tmp>/doc.json" "<skill>/cache/<slug>.doc.json"
python3 "<skill>/scripts/build.py" "<skill>/cache/<slug>.doc.json" "<skill>/cache/<slug>.html"
```
같은 플랜을 다시 그리면 같은 슬러그로 덮어써 캐시가 갱신된다. 검증 실패(미지의 블록 타입, meta.title
누락 등) 시 메시지대로 `doc.json`을 고쳐 다시 빌드한다.

### Step 5 — 열기 (선택)
생성 파일 경로를 사용자에게 알린다: `<skill>/cache/<slug>.html`. self-contained라 파일을 그대로 브라우저로
열면 된다. 자동으로 열려면 OS 무관하게 python3 표준 라이브러리를 쓴다(macOS 전용 `open` 대신):
```bash
python3 -c "import webbrowser,sys; webbrowser.open('file://'+sys.argv[1])" "<skill>/cache/<slug>.html"
# 셸 대안: Linux=xdg-open · macOS=open · Windows=start
```

## 참고
- 결과물은 self-contained · 다크/라이트 자동 대응(보는 사람 테마, 우상단 토글) · 반응형. 코드/JSON 블록은
  Dracula 테마로 문법 하이라이트된다(build.py가 토큰화).
- 출력은 `<skill>/cache/<slug>.html`에 누적된다(캐시). 같은 슬러그면 덮어쓴다. 게시 도구로 올리지 않음(파일만).
- **레이아웃**: 상단 hero(한눈 요약, 항상 펼침) + 각 level-2 섹션은 접이식(`<details>`, 기본 접힘). 상단
  목차에 "모두 펼치기" 토글이 있고, 목차 링크 클릭 시 해당 섹션이 자동으로 열린다.
- 엔드포인트 카드도 `<details>`로 접힌다(처음부터 펼치려면 `"open": true`). 파일 스코프는 `tree`(변경배지
  신규/수정/삭제/재생성), 규모 수치는 `stats` 칩으로 — 이 배선은 build.py/템플릿이 자동 처리한다.
- 병렬 팬아웃의 벽시계 시간 ≈ 가장 느린 섹션 1개. 섹션이 균등하게 나뉘도록 크기를 맞춘다.
- 런타임 이식: build.py·template.html은 python3만 있으면 그대로 동작(툴명 비의존). 지침에서 특정 툴 이름을
  가정하지 말고 "파일을 읽는다/쓴다", "하위 에이전트로 위임한다"처럼 행위로 이해하라.
- 코드 구조(코드베이스) 시각화가 필요하면 architecture-flow-map을 쓴다. 이 스킬은 계획 문서 전용.
