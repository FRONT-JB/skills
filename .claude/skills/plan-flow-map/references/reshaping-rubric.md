# 재구성 규칙 (읽기 쉽게 — 일관성 + 무결성)

여러 서브에이전트가 각자 맡은 섹션을 블록으로 바꿔도 **(1) 결과가 일관되고 (2) 원문에서 빠지는 내용이
없도록** 하는 결정 규칙이다. 모든 서브에이전트는 이 파일을 읽고 그대로 따른다. "읽기 쉽게"는 표현·배치·
강조를 다듬는 것이지 **내용을 삭제·요약해 없애는 것이 아니다.**

## 1. 콘텐츠 → 블록 매핑 (이 표를 그대로 따른다 · 같은 형태는 항상 같은 블록)

| 원문 형태 | 블록 | 비고 |
|---|---|---|
| 섹션 제목 (`##`) | `heading` level 2 | 섹션의 **첫 블록**. 제목 텍스트는 원문 그대로 |
| 하위 제목 (`###` 이하) | `heading` level 3 | |
| 섹션 도입/요지 문장 | `prose` (한 줄 리드) | heading 바로 다음. "이 섹션이 무엇인지" 1문장 |
| 일반 문단 | `prose` | 3문장 이하로 분할. 식별자·경로·명령은 `` `code` `` 인라인 |
| 불릿 목록 | `bullets` | 항목 텍스트 보존 |
| 번호/순서 목록 | `bullets`(ordered) 또는 `table` | 열이 여럿(순서·작업·책임 등)이면 `table` |
| 명령/경로 나열 | `bullets`(mono) 또는 `code` | 명령은 verbatim |
| 현재↔목표 / as-is↔to-be / 대안 비교 | `compare` | 각 열에 원문 항목 그대로 |
| 마크다운 표 (진짜 속성 격자) | `table` | **모든 행·열 보존**. 단 아래 "파일 목록" 예외 |
| **파일·경로 목록** (소스가 표여도) | `tree` | 경로 한 열 + 변경종류/설명은 표가 아니라 트리. `change` 배지로 신규/수정/삭제/재생성 |
| API (METHOD /path …) | `endpoint` | method·path·params·request·response·설명 |
| 데이터 모델/스키마/타입(필드+타입) | `entity` | **모든 필드·타입·관계** |
| 디렉토리/파일 구조 | `tree` | **모든 경로** |
| 코드펜스 (```) | `code` | 언어 라벨만, 내용 verbatim |
| 목표 / goal / objective | `callout` tone=`goal` | |
| 게이트 / 완료조건 / acceptance / done when / 통과 기준 | `callout` tone=`gate` | 실제 명령·기준 명시 |
| 경고 / 주의 / 위험 / risk / pitfall | `callout` tone=`warn` | |
| 참고 / note / tip | `callout` tone=`info` | |

**형식보다 의미 우선.** 소스의 마크다운 형식(표/불릿 등)이 아니라 **내용의 의미**로 블록을 고른다.
가장 흔한 실수: 파일 목록이 마크다운 표로 쓰였다고 `table`로 옮기는 것 — 파일·경로 목록은 **`tree`**다.
표에 딱 맞지 않는 형태는 **가장 가까운 행**을 고른다. 새 블록 타입을 지어내지 않는다(`doc-schema.md`의
11종만 사용).

## 2. 무손실 — 정보는 남기고 표현은 조인다 (가장 중요)

핵심 구분: **무손실 = 모든 "항목·사실"을 남기는 것**이지, 원문 문장을 그대로 옮기는 게 아니다.
문장 길이(표현)는 오히려 최대한 압축한다. **"정보 보존 ≠ 문장 보존".**

- **모든 원자 단위**(리스트 항목 · 표의 각 행 · 코드/명령 · 엔드포인트 · 필드 · 파일 경로 · 게이트 ·
  제약 · 불변식 단계)는 **정확히 하나의 블록에 반영**한다. 항목을 삭제·병합해 없애지 않는다.
- **기술 토큰은 verbatim**: 식별자 · 파일 경로 · 명령 · 숫자 · 상태값 · 타입 · URL. 절대 의역·변형 금지.
- **산문 표현은 압축한다**(§3): 원문의 긴 문단을 복붙하지 말고 요지로 재서술한다 — 단, 그 안의
  항목·사실·토큰은 하나도 잃지 않는다.
- 애매하면 **항목은 포함**한다(완전성 우선). **원문 등장 순서 유지**.

## 3. 표현 압축 — 짧고 스캔되게 (산문 벽 금지)

읽는 길이를 줄이는 게 목표다. 정보를 지우는 게 아니라 문장을 조인다. 아코디언(상세)도 이 규칙을 따른다.

- **산문 벽 금지.** `prose` 블록은 **최대 2문장**. 배경·이유의 장황한 서술은 요지 한 줄로.
- **나열은 무조건 `bullets`/`table`.** "A하고 B하고 C한다" 식 문장은 항목별 불릿으로 쪼갠다.
- **각 세부 블록/서브섹션은 한 줄 리드로 시작**하고, 세부는 불릿·표로 내린다.
- **군더더기 제거**: 반복 설명, "~하기 때문에 …이다" 장식, 접속사 나열을 걷어내고 요지 + 근거만.
- 문장의 **핵심 결정·용어는 볼드**, 식별자·경로·명령은 `` `code` ``.
- 제약/위험/게이트는 문단에 묻지 말고 `callout`으로 끌어올린다.
- 평이하고 능동적인 말투. 관료적 표현·불필요한 수동태 지양.

## 4. 쓰기 전 자가 점검 (각 서브에이전트가 파일 저장 직전에 확인)

- [ ] 담당 섹션의 **모든** 항목·표 행·코드·엔드포인트·필드·경로가 블록에 들어갔는가? (누락 0)
- [ ] 기술 토큰을 verbatim으로 옮겼는가? (경로·명령·숫자·타입)
- [ ] **표현이 원문보다 짧아졌는가?** 산문 벽 없이 문단 최대 2문장, 나열은 불릿/표로 조였는가?
- [ ] §1 매핑 표를 따랐는가? (같은 형태 → 같은 블록)
- [ ] 섹션 heading 다음에 한 줄 리드가 있는가? 원문 순서를 유지했는가?
